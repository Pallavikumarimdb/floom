## Executive summary
- Connections is the user-side credential layer for Floom.
- It is parallel to, and separate from, v0.1 app-owner secrets.
- A Floom user connects Gmail, Slack, Linear, Notion, and GitHub once from a Floom settings page.
- Floom uses one platform Composio account to broker OAuth, token refresh, and connected-account lifecycle.
- Apps declare required provider actions in manifest metadata.
- Floom asks the end user for app-specific consent before an app can use those connected accounts.
- Runtime execution resolves the calling user, checks consent, resolves the appropriate connected account, and exposes a safe execution-scoped interface inside the sandbox.
- Floom must not inject the long-lived platform `COMPOSIO_API_KEY` directly into app sandboxes.
- Capability K partially works before capability G only for short synchronous flows that fit inside the current 60-second execution budget.
- Full "bulk inbox / bulk workspace / long-running agent" stories remain gated on capability G async + poll.
## Product definition
Connections means:
- Floom owns the Composio project.
- Floom creates and manages the platform auth configs.
- Floom users authenticate to providers through Composio-hosted Connect Links launched from Floom.
- Floom stores only provider/account mapping metadata, never raw provider tokens.
- Floom apps declare a curated provider-action subset in manifest metadata.
- Floom asks for user consent per app before an app can use a connected account.
- Floom runtime executes those actions on behalf of the current user with strong scoping and auditability.
Connections does not mean:
- App authors receive the raw Composio platform API key.
- App authors receive raw provider OAuth tokens.
- Floom exposes the entire Composio tool surface directly in manifests.
- Anonymous callers can use user-side integrations.
- Private-app sharing semantics are widened in this capability.
## Goals
### Primary goals
- Let a Floom user connect a provider once at the platform level.
- Let multiple apps reuse that same connection with per-app consent.
- Keep manifests stable and auditable by exposing only a curated action subset.
- Keep raw provider credentials outside Floom storage.
- Keep the platform Composio API key outside untrusted app sandboxes.
- Preserve the existing v0.1 public/private app model.
### Non-goals
- Full Composio meta-tool exposure inside app manifests.
- Arbitrary Composio tool slug pass-through from manifests.
- Enterprise admin tools such as Salesforce, HubSpot, or Stripe in phase 1.
- Organization-wide delegated impersonation.
- Anonymous usage of connected providers.
- Bulk long-running workflows that rely on capability G but try to masquerade as synchronous v0.1 runs.
## Dependency on capability G
Capability K depends on capability G for the complete product promise.
That dependency is real.
It is not absolute for all phase-1 use cases.
### What works before capability G
The following are explicitly in scope before async + poll lands:
- Gmail send-email actions that complete within one request.
- Gmail list first-page threads for a small inbox slice.
- Slack post-message.
- Slack add-reaction.
- Linear create-issue.
- Linear comment on issue.
- Notion create-page.
- Notion append small blocks to a page.
- GitHub create-issue.
- GitHub comment on issue.
- GitHub read a single file.
### What does not work well before capability G
The following are not reliable inside the current 60-second sandbox model:
- Summarize hundreds of unread emails across many pages.
- Traverse large Slack histories or many channels.
- Sync an entire Notion workspace.
- Run wide GitHub repository analysis over many files and pull requests.
- Any provider workflow that needs retries over rate-limit windows longer than a single request.
### Required product honesty
- The settings page copy may say "Connect once for supported apps."
- It must not say "all workflows" or imply bulk agent jobs are fully supported before G.
- `/docs` and `get_app_contract` must spell out that v0.x Connections without async is for bounded synchronous flows only.
## High-level design
The capability is implemented as five layers:
1. Provider catalog
2. Connection storage and lifecycle
3. Manifest declaration and consent
4. Runtime connection resolution
5. Safe Composio execution interface
### Key design call
The platform Composio API key stays server-only.
This spec rejects direct injection of a long-lived `COMPOSIO_API_KEY` into app sandboxes.
That rejection is required because otherwise:
- Any published app could exfiltrate the platform key.
- Any published app could attempt calls outside its declared integration subset.
- Floom could not reliably meter per-action usage.
- Floom could not enforce scope tiering or connected-account scoping at the server boundary.
Instead:
- The sandbox receives only execution-scoped Floom credentials.
- The sandbox calls a Floom runtime proxy for curated provider actions.
- The proxy uses the server-side Composio API key and the resolved connected-account mapping.
This keeps the app-author experience simple without leaking a platform secret.
## Invariants
The implementation must satisfy all of the following:
- No raw provider OAuth tokens are stored in Floom tables.
- No raw provider OAuth tokens are returned to browsers, MCP, or sandboxes.
- No raw Composio platform API key is exposed to sandboxes.
- Anonymous callers never gain access to user-side providers.
- App-owner secrets and user-side connections remain separate systems.
- Manifest shorthand must expand to a frozen per-version action set.
- Per-app user consent defaults to deny.
- Provider access is bounded by:
  - declared manifest integrations
  - user's connected accounts
  - user's app-specific consent
  - execution-scoped proxy token
- Public metadata visibility and public run visibility are separate concepts once integrations are involved.

## Catalog strategy
Floom exposes a curated action subset per provider.
Floom does not expose raw Composio toolkit slugs in manifests.
Reasons:
- Manifest stability
- Consent readability
- Auditable allowlist
- Easier testing
- Lower blast radius when Composio adds or renames tools
The provider catalog is a Floom-owned source of truth.
The implementation must keep one catalog record per action with:
- provider
- Floom action id
- human label
- action class
- minimum scope tier
- current Composio toolkit slug
- current pinned toolkit version
- sync suitability
- output risk notes
That catalog belongs in a server-side module, not in user manifests.
## Catalog stability rules
- Floom action ids are the public contract.
- Composio tool slugs are internal implementation data.
- Adding a new curated action is additive only.
- Removing or changing an existing curated action requires a new capability spec and migration note.
- Provider shorthand in manifests expands at publish time, not at run time.

## Provider-wide policy decisions
### One curated subset, not full Composio
The public manifest contract is:
- provider ids
- Floom action ids
The public manifest contract is not:
- raw Composio tool slugs
- arbitrary toolkit wildcard access
- direct provider API endpoint pass-through
### Provider shorthand semantics
If a manifest says:
- `gmail`
- `slack`
- `linear`
- `notion`
- `github`
then publish-time normalization expands that shorthand to the full current curated subset for that provider.
That expansion is frozen into the published app version.
It does not float as the catalog changes later.
This avoids silent permission broadening for already-published apps.
### Curated subset governance
Every new curated action must ship with:
- catalog entry
- consent copy
- proxy schema
- scope-tier classification
- rate-limit notes
- integration test evidence
## Manifest contract
## New top-level field
Add a new optional top-level manifest field:
- `integrations`
This field is:
- optional
- additive to v0.1
- invalid if not an array
- invalid if the array is empty
Apps without `integrations` behave exactly like current v0.1 apps.
## Allowed manifest tokens
Each `integrations` item is one of four forms:
1. `provider`
2. `provider.action`
3. `provider:alias`
4. `provider:alias.action`
Where:
- `provider` is one of `gmail`, `slack`, `linear`, `notion`, `github`
- `action` is one curated Floom action id valid for that provider
- `alias` is a user-visible account alias slug
### Why alias exists in the grammar
Multiple connected accounts per provider are a first-class feature.
Examples:
- two Gmail accounts
- two Slack workspaces
- work and personal GitHub accounts
The manifest needs a stable way to pin an app to a specific connected account when the app contract requires that.
## Alias grammar
Aliases must satisfy:
- lowercase letters, numbers, hyphens
- 2 to 32 chars
- regex: `^[a-z0-9][a-z0-9-]{0,30}[a-z0-9]$`
Aliases are user-owned labels.
Aliases are not provider ids.
Aliases are not globally unique.
Aliases are unique only within:
- one user
- one provider
## Manifest examples that must be rejected
- `integrations: []`
- `integrations: [salesforce]`
- `integrations: [gmail.delete_thread]`
- `integrations: [gmail:Work.send_email]`
- `integrations: [gmail.send_email, gmail.send_email]`
- `integrations: [gmail:work, gmail]`
- `integrations: [slack.post_message, slack:acme.post_message, slack:acme.post_message]`
## Normalization rules
The validator must normalize each manifest into three published forms:
1. `integrations_declared`
2. `integrations_resolved`
3. `integrations_summary`
### `integrations_declared`
- exact normalized token strings from the manifest
- deduplicated
- sorted
- preserved on the app version
### `integrations_resolved`
- expanded atomic action requirements
- each item is one canonical atom:
  - `provider.action`
  - `provider:alias.action`
- sorted
- deduplicated
- frozen per app version
### `integrations_summary`
- grouped display shape for UI and API
- example:
  - Gmail: `send_email`, `list_threads`
  - Slack: `post_message`
## Validator behavior
The manifest validator in `src/lib/floom/manifest.ts` must:
- add `integrations` to accepted keys
- reject unknown providers
- reject unknown actions
- reject malformed aliases
- reject duplicate declared tokens
- reject duplicate resolved atoms after expansion
- reject mutually ambiguous declarations
### Ambiguity rules
The validator must reject ambiguous combinations such as:
- both `gmail` and `gmail.send_email`
- both `slack:acme` and `slack:acme.post_message`
- both `github.read_file` and `github:work.read_file` when the manifest otherwise depends on one default account
Reason:
- The app contract must be explicit.
- Consent copy must be deterministic.
- Runtime binding rules must be simple.
## Manifest storage changes
The published app version must store the resolved integration contract.
This requires new columns on `public.app_versions`.
### New `app_versions` columns
| Column | Type | Null | Purpose |
|---|---|---|---|
| `integrations_declared` | `text[]` | not null default empty | manifest-level declared tokens |
| `integrations_resolved` | `text[]` | not null default empty | atomic publish-time snapshot |
| `integrations_summary` | `jsonb` | not null default `{}` | grouped UI/API shape |
### Why app versions own this data
- permissions are part of the versioned run contract
- shorthand expansion must be frozen at publish time
- consent checks must compare against the version being executed
## Public metadata changes
`GET /api/apps/:slug` must return:
- `integrations_declared`
- `integrations_resolved`
- `integrations_summary`
- `requires_sign_in_for_run`
For apps with `integrations`, `requires_sign_in_for_run` is always true.
Even if `public: true`.
Reason:
- the app page may stay publicly viewable
- the run path cannot stay anonymous
## Storage model
## New tables
Add three new tables:
1. `user_connections`
2. `app_user_consents`
3. `connection_action_events`
## Table: `user_connections`
### Purpose
- store Floom's mapping from a user to a connected provider account
- track status, alias, scope tier, and last use
- avoid storing raw provider credentials
### Columns
| Column | Type | Null | Notes |
|---|---|---|---|
| `id` | `uuid` | not null | primary key |
| `user_id` | `uuid` | not null | FK to `auth.users(id)` |
| `provider` | `text` | not null | one of `gmail`, `slack`, `linear`, `notion`, `github` |
| `account_alias` | `text` | not null | user-facing stable alias slug |
| `account_label` | `text` | not null | display label, e.g. email address or workspace name |
| `provider_account_ref` | `text` | null | provider-native stable ref when available, e.g. Slack team id |
| `composio_entity_id` | `text` | not null | Floom-owned stable Composio user/entity id for the Floom user |
| `composio_connected_account_id` | `text` | not null | Composio connected account id |
| `scope_tier` | `text` | not null | provider-specific tier id |
| `is_default` | `boolean` | not null default false | one default per user/provider |
| `status` | `text` | not null | `active`, `reconnect_required`, `disconnected`, `errored` |
| `status_reason` | `text` | null | last known provider or Composio failure code |
| `connected_at` | `timestamptz` | not null | connection established time |
| `disconnected_at` | `timestamptz` | null | soft-delete / revoke marker |
| `last_used_at` | `timestamptz` | null | updated on successful runtime use |
| `created_at` | `timestamptz` | not null | default now |
| `updated_at` | `timestamptz` | not null | default now |
### Important clarifications
- `composio_entity_id` is the stable user-scoped identifier Floom uses inside Composio.
- In practice Floom can derive it deterministically from the Floom user id.
- `composio_connected_account_id` is still required because Composio uses connected-account selection for multi-account flows.
- The runtime must prefer `connected_account_id` when a specific account binding is required.
### Unique constraints
Add the following constraints:
- primary key on `id`
- unique on `composio_connected_account_id`
- unique active alias per `(user_id, provider, account_alias)` where `disconnected_at is null`
- unique default per `(user_id, provider)` where `is_default = true and disconnected_at is null`
### Check constraints
- provider allowlist check
- status allowlist check
- alias format check
- default/disconnected consistency check:
  - disconnected rows cannot remain default
### Why soft delete is required
Soft delete via `status` + `disconnected_at` is preferred over hard delete because:
- audit trails matter
- reconnect UX can reuse metadata
- future billing and support investigations need history
The UI may say "disconnect" and remove it from the active list.
The DB record remains with `status = disconnected`.
The brief requested "delete the row".
This spec intentionally chooses soft deletion instead because it is safer operationally.
## Table: `app_user_consents`
### Purpose
- store explicit per-app-per-user grant state
- track what resolved integration atoms the user approved
- make re-prompt logic deterministic across app versions
### Columns
| Column | Type | Null | Notes |
|---|---|---|---|
| `id` | `uuid` | not null | primary key |
| `app_id` | `uuid` | not null | FK to `apps(id)` |
| `user_id` | `uuid` | not null | FK to `auth.users(id)` |
| `granted_integrations` | `text[]` | not null | sorted atomic set |
| `granted_via` | `text` | not null | `permalink`, `api`, `mcp`, `settings` |
| `granted_at` | `timestamptz` | not null | initial grant time |
| `revoked_at` | `timestamptz` | null | null when active |
| `last_used_at` | `timestamptz` | null | updated when a run uses the consent |
| `created_at` | `timestamptz` | not null | default now |
| `updated_at` | `timestamptz` | not null | default now |
### Unique constraints
- unique active consent row per `(app_id, user_id)` where `revoked_at is null`
### Consent semantics
- default deny
- app access allowed only when requested integration atoms are a subset of `granted_integrations`
- if requested atoms exceed grant, prompt again and replace the active row with a merged grant set
### Revocation semantics
- revoking app consent does not disconnect the provider account
- revoking app consent only breaks that app's access to the connection
## Table: `connection_action_events`
### Purpose
- track per-app-per-user-per-action usage
- support future billing
- support auditing and abuse handling
### Columns
| Column | Type | Null | Notes |
|---|---|---|---|
| `id` | `uuid` | not null | primary key |
| `execution_id` | `uuid` | not null | FK to `executions(id)` |
| `app_id` | `uuid` | not null | FK to `apps(id)` |
| `version_id` | `uuid` | null | FK to `app_versions(id)` |
| `user_id` | `uuid` | not null | end user whose account was used |
| `provider` | `text` | not null | provider id |
| `account_alias` | `text` | null | alias used for the action |
| `connection_id` | `uuid` | not null | FK to `user_connections(id)` |
| `integration_atom` | `text` | not null | e.g. `gmail.send_email` |
| `status` | `text` | not null | `success`, `error`, `rate_limited`, `reconnect_required`, `denied` |
| `provider_error_code` | `text` | null | upstream error identifier |
| `duration_ms` | `integer` | null | measured proxy-side |
| `created_at` | `timestamptz` | not null | default now |
### Why this table is needed
If Floom lets app code call Composio directly with the platform key:
- Floom cannot reliably meter per action.
Because this spec routes execution through a Floom proxy:
- per-action events can be recorded server-side.
That is required for future billing and quota enforcement.

Add optional columns to `public.executions`:
| Column | Type | Null | Purpose |
|---|---|---|---|
| `effective_user_id` | `uuid` | null | the user whose connections were used |
| `requires_integrations` | `boolean` | not null default false | quick filter |

## RLS policy requirements
## `user_connections` RLS
Enable RLS.
Policies:
- owner can select own rows
- owner can insert own rows
- owner can update own rows
- owner can delete own rows only if implementation chooses hard delete
- no anon policy
- no cross-user read policy
Important:
- public apps must not expose `user_connections` through any anon Supabase path
- public app execution remains mediated by server routes using service role
## `app_user_consents` RLS
Enable RLS.
Policies:
- user can read own rows
- user can create own rows
- user can update own rows
- no anon policy
## `connection_action_events` RLS
Enable RLS.
Policies:
- the end user may read rows where `user_id = auth.uid()`
- the app owner may read rows for their own apps
- no insert/update/delete policy for user role
- writes happen through service role only
## Index requirements
Add indexes on:
- `user_connections(user_id, provider, status)`
- `user_connections(user_id, provider, is_default)`
- `app_user_consents(app_id, user_id, revoked_at)`
- `connection_action_events(execution_id)`
- `connection_action_events(app_id, user_id, created_at desc)`
- `connection_action_events(provider, created_at desc)`
## Composio user id model
Floom owns one Composio project.
Each Floom user maps to one stable Composio user/entity id.
Use:
- deterministic id format: `floom_user_<supabase-user-uuid>`
Reasons:
- no extra lookup table required
- easy idempotency
- safe to recompute
The `composio_entity_id` column still remains in `user_connections` because:
- the brief requested it
- it aids support and auditing
- it future-proofs against any id format migration
## User-facing connection flow
## Route choice
Use:
- page route: `/settings/connections`
Do not use:
- `/me/connections`
Reason:
- existing product copy already has settings-oriented pages such as `/tokens`
- this is a user settings concern, not a public profile concern
## Page layout
The page shows one card per supported provider.
Each card contains:
- provider name
- short description
- connection status badge
- connected account rows for that provider
- connect button
- disconnect button on active accounts
- reconnect button for `reconnect_required`
- upgrade scopes button when an app flow requests a higher tier
- "default" badge and "make default" action
Card states are `Not connected`, `Connected`, `Reconnect needed`, and `Multiple accounts`. Connected rows show label, alias, scope tier, last used, default status, and the appropriate connect/disconnect/reconnect/default actions.
## Entry points into the flow
Users may reach connection UX from:
- `/settings/connections`
- `/p/<slug>` preflight card
- API error response CTA
- MCP `connect_provider`
All entry points converge on the same server-side connect-link creation endpoint.
## API routes for connection management
Add:
- `GET /api/me/connections`
- `POST /api/me/connections/:provider/connect`
- `GET /api/me/connections/callback`
- `POST /api/me/connections/:provider/:connectionId/disconnect`
- `POST /api/me/connections/:provider/:connectionId/reconnect`
- `POST /api/me/connections/:provider/:connectionId/default`
- `PATCH /api/me/connections/:provider/:connectionId`
Route semantics:
- `GET /api/me/connections` returns supported providers, active/disconnected rows, scope tier, reconnect state, and default-account state.
- `POST /api/me/connections/:provider/connect` accepts optional `requested_atoms`, `requested_scope_tier`, `suggested_alias`, `return_to`, and `allow_multiple`, then returns `redirect_url` and `expires_at`.
- `GET /api/me/connections/callback` validates state, resolves the connected-account result, upserts `user_connections`, and redirects back to Floom.
- `POST /api/me/connections/:provider/:connectionId/disconnect` revokes or disables the upstream connection and marks the row disconnected.
- `POST /api/me/connections/:provider/:connectionId/reconnect` creates a new connect link for the same alias and preserves row continuity until success.
- `POST /api/me/connections/:provider/:connectionId/default` atomically switches the default account for that provider.
- `PATCH /api/me/connections/:provider/:connectionId` can rename alias and user-editable display label only; provider-native identity is immutable.
## State parameter requirements
The connection callback must use a signed, tamper-evident state token.
State must include:
- user id
- provider
- requested scope tier
- suggested alias if any
- return path
- nonce
- issued at
- expiry
TTL:
- 15 minutes
Reason:
- avoid extra in-flight request table
- keep callback correlation deterministic
- protect against CSRF and open redirect abuse
## Callback completion behavior
Composio manual-auth flows may return before the connected account is fully active.
The callback path must handle three cases:
1. active immediately
2. initiated but not active yet
3. failed / denied
Handling:
- if active immediately, fetch metadata, derive alias/label, upsert the row, and redirect with success toast
- if still initiated, show a Floom finishing page that polls `GET /api/me/connections` every 2 seconds for up to 30 seconds
- if failed or denied, do not create an active row and redirect back with error toast
## Alias derivation rules
When a connection first lands, derive:
- `account_label`
- `account_alias`
Alias sources:
- Gmail uses email address
- Slack uses workspace name
- Linear uses workspace or team display name
- Notion uses workspace name
- GitHub uses login name
If an alias collides for the same user/provider, append a numeric suffix and let the user edit it on the return screen.
## Default-account rules
Defaults are per user and provider.
Rules:
- first active connection for a provider becomes default automatically
- if the default is disconnected, choose another active connection if exactly one exists
- if several remain, clear default and force the user to pick next time

## Agent token semantics
Current verified state from repo inspection:
- `agent_tokens` has no `acting_as` column
- `resolveAuthCaller()` resolves agent token calls as token owner
Therefore phase-1 Connections semantics are:
- user JWT caller runs as that user
- agent token caller runs as token owner

## `as_user` decision
This spec still adds `as_user` to REST and MCP contracts for forward compatibility.
Phase-1 rule:
- if `as_user` is absent, effective user is caller user
- if `as_user` equals caller user, accept
- if `as_user` differs from caller user, return 403 `delegation_not_supported`
This is the safest rule because:
- it is truthful to the current schema
- it does not invent hidden delegation state
- it leaves the contract forward-compatible
Any broader delegation model is explicitly out of scope for capability K.

## Runtime resolution flow
## Effective-user resolution
Resolve in order:
1. authenticate caller
2. derive caller user id
3. validate `as_user`
4. set effective user id
If no caller and integrations required:
- fail with 403 before sandbox execution
## Requested-binding resolution
Resolve provider bindings in order:
1. manifest-pinned alias
2. request `connection_bindings`
3. user default connection
4. single active connection fallback
If still unresolved:
- fail with `connections_binding_required`
## Consent check
Compute:
- version's `integrations_resolved`
Check:
- active consent row exists
- requested atoms are subset of granted atoms
If not:
- fail before sandbox execution
## Connection check
For each provider atom needed:
- find active `user_connections` row for the resolved alias or chosen default
- verify `status = active`
- verify scope tier covers required action
If any check fails:
- fail before sandbox execution
## Composio execution model
The server creates one execution-scoped proxy grant.
That grant is bound to:
- execution id
- app id
- app version id
- effective user id
- allowed integration atoms
- resolved connection ids
- expiry
TTL:
- execution timeout plus 2 minutes buffer
The sandbox uses that grant to call a Floom runtime proxy.
## Why direct Composio SDK injection is rejected
The task brief proposed:
- `COMPOSIO_API_KEY`
- provider-specific entity ids
This spec rejects that for production because:
- sandboxes run untrusted user code
- a long-lived platform key is too sensitive
- one compromised app would compromise the entire Composio project
The runtime still mirrors `runtime-secrets.ts` conceptually:
- server-side resolution
- execution-scoped injection
- no client exposure
But the injected secret is a Floom-scoped execution token, not the platform Composio key.

## Runtime proxy API
Add internal route:
- `POST /api/runtime/connections/execute`
Auth:
- bearer `FLOOM_COMPOSIO_EXECUTION_TOKEN`
Request body:
- `integration`
- `arguments`
Example conceptual integration values:
- `gmail.send_email`
- `slack.post_message`
- `linear.create_issue`
- `notion.create_page`
- `github.create_issue`
Server behavior:
1. validate execution token
2. validate integration atom is allowlisted for that execution
3. resolve connection row
4. map Floom action id to Composio tool slug
5. call Composio with:
   - server-side API key
   - effective Composio user/entity id
   - resolved `connected_account_id`
6. record `connection_action_events`
7. return upstream result

## Security and privacy
## What Floom does and does not store
Floom does store:
- connection metadata
- provider aliases
- consent grants
- execution outputs
- action event logs
Floom does not store:
- raw provider OAuth tokens
- raw Composio credentials
## Important privacy clarification
The sentence "Floom never sees user data" is not accurate enough for this product.
Accurate statement:
- Floom never stores the raw provider credentials.
- Provider data may still pass through the app run pipeline and may be stored in `executions.output` if the app returns it.
That is already true for any other app output today.
Therefore `/settings/connections` privacy copy must say:
- `Floom brokers the connection and does not store the provider OAuth token.`
- `Provider data requested by an app can still flow through that app's execution and output, just like any other app input or result.`
## Secret boundary summary
Three separate credential layers exist after K:
1. Supabase session / agent token auth
2. app-owner secrets from `app_secrets`
3. user-side provider access via Composio
They must never be conflated in copy or in runtime code.
## Logging rules
Never log:
- provider access tokens
- Composio platform key
- full connect-link URLs with reusable secrets
- provider payloads unless explicitly part of app output persistence
May log:
- provider id
- connection id
- account alias
- action atom
- provider error code
- execution id
## Abuse handling
Rate limits exist at three layers:
- Floom public app run rate limits
- Floom runtime-proxy action limits
- Composio / provider upstream quotas
Phase 1 must implement:
- per execution max 25 connection actions
- per action request-body size cap
- per user per hour provider-action cap configurable by env
Reason:
- one app run must not fan out uncontrollably inside the sandbox
## API error contract
All connection-related run errors must follow one envelope shape:
```json
{
  "error": "human readable message",
  "error_code": "machine_code",
  "required_integrations": ["gmail.send_email"],
  "missing_providers": ["gmail"],
  "reconnect_providers": [],
  "consent_url": "/p/example?consent=1",
  "connect_url": "/settings/connections?next=/p/example",
  "sign_in_url": "/login?next=/p/example"
}
```
Fields may be omitted when not relevant.

## Final implementation decisions summary
### Accepted
- one platform Composio account
- platform-managed connection UI at `/settings/connections`
- three new DB tables
- manifest `integrations`
- publish-time shorthand expansion frozen per app version
- per-app-per-user consent
- multiple accounts per provider
- alias pinning
- connection-binding selection
- runtime proxy instead of raw platform key injection
- phase-1 self-only `as_user`
### Rejected
- direct sandbox access to long-lived `COMPOSIO_API_KEY`
- anonymous runs for integration-backed apps
- full Composio action surface in manifests
- hard delete as the only disconnect behavior
- implicit consent from provider connection alone
- silent delegation for agent tokens
