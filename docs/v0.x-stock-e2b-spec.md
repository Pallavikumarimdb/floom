# v0.x Stock-E2B Mode Spec

Branch: `feat/v0.x-stock-e2b`

This branch widens Floom from a narrow single-file Python contract to a thin wrapper over stock E2B base. Runtime depth belongs to E2B. Floom keeps only the constraints needed for sharing, secrets, redaction, and the one-run contract.

## Goals

- Publish and run full app directories as tarballs.
- Accept command-based manifests with optional schemas.
- Preserve the v0.1 manifest contract unchanged for old apps.
- Keep the current synchronous 60-second cap while async stays separate.

## Manifest modes

### Preferred stock-E2B manifest

```yaml
slug: my-app
input_schema: ./input.schema.json
output_schema: ./output.schema.json
secrets: [GEMINI_API_KEY]
public: true
```

Optional:

- `command: <shell>`
- `dependencies.python: ./requirements.txt --require-hashes`
- `bundle_exclude: [...]`
- `name`, `description`

### Legacy v0.1 manifest

```yaml
runtime: python
entrypoint: app.py
handler: run
```

Still accepted as-is.

## Command detection

When `command` is omitted:

- `app.py` -> `python app.py`
- `index.js` -> `node index.js`
- `package.json` with `scripts.start` -> `npm start`
- multiple matches fail with `ambiguous command auto-detection`
- zero matches fail

`main.go` auto-detection is intentionally not enabled in this branch because stock-base Go availability was not re-verified inside this workspace.

## Bundle format

- Storage bucket remains `app-bundles`
- New path shape: `<owner_id>/<slug>/<uuid>.tar.gz`
- Content type: `application/gzip`
- `app_versions.bundle_kind` distinguishes `single_file` and `tarball`

### Upload validator

Reject when:

- compressed size > `5 MB`
- unpacked size > `25 MB`
- file count > `500`
- any file > `10 MB`
- decompression ratio > `100x`
- path traversal or absolute paths
- symlink, hardlink, device file, or FIFO entries
- `floom.yaml` missing at tarball root
- no command can be resolved

## Runtime flow

1. Download bundle from Storage.
2. If `bundle_kind = single_file`, materialize legacy entrypoint in `/home/user/`.
3. If `bundle_kind = tarball`, extract to `/home/user/app`.
4. If `requirements.txt` exists, run `pip install -r requirements.txt`.
5. If `package.json` exists, run `npm install`.
6. Inject secrets as env vars.
7. Inject the same JSON inputs to stdin and `FLOOM_INPUTS`.
8. Run the command.
9. Return:
   - validated JSON when `output_schema` exists
   - parsed JSON when stdout final line is JSON and no schema exists
   - `{ stdout, exit_code }` with a 4 KB stdout tail otherwise

## Output and failure envelopes

Failure responses use one shared shape across browser, REST, and MCP:

- install failure: `status: failed`, `error.phase: install`
- non-zero exit: `status: failed`, `error.phase: run`
- timeout: `status: timed_out`, `error.phase: run`
- output validation failure: `status: failed`, `error.phase: output_validation`
- sandbox boot failure: HTTP `502`, `error: sandbox_unavailable`

## Secrets and static coverage check

At publish time the validator scans `.py`, `.js`, and `.ts` files for common env-var access patterns and warns when a referenced env var is not:

- declared in `secrets:`
- a `FLOOM_*` internal var
- in the small public allowlist (`PATH`, `HOME`, `PORT`, and similar)

Warnings are returned with the publish response. They do not block publish.

## Abuse and cost controls

Existing:

- anonymous caller: `20` runs / `60s`
- app-wide public: `100` runs / `60s`
- sync timeout: `60s`

New:

- per app: `30` E2B minutes / day
- per owner: `2` E2B hours / day across all apps
- tracked in `app_quota_usage`
- failures count too

Out of scope:

- E2B network egress governance
- owner misuse of their own BYOK provider keys
- arbitrary HTTP route proxying

## Backward compatibility

- old rows stay `bundle_kind = single_file`
- legacy manifests still validate
- legacy apps keep running
- new publishes of legacy manifests go through the tarball path
